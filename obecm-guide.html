Section 1.0

On-Board Energy Consumption Monitoring (OBECM)

1.1

Overview

On-Board Energy Consumption Monitoring, or OBECM, is a vehicle feature that collects energy-consumption-related data, organizes it into a defined data set, makes it available through diagnostics, and sends it to the cloud through telematics.

⚡
Simple Definition

OBECM is a vehicle data feature that records key energy-use information and sends a periodic snapshot of that information outside the vehicle.

For battery electric vehicles, OBECM focuses on energy-related values such as distance traveled, energy used, battery-related values, and energy charged into the battery.

ℹ️
Not a Driver-Operated Feature

The customer does not turn OBECM on, configure it, or interact with it directly. The customer simply uses the vehicle normally, and the feature collects the required information in the background.

OBECM is also not the main owner of deep scientific correctness of every energy-related value. OBECM's main role is to ensure that the required data is:

  • Present
  • Structured correctly
  • Available through diagnostics
  • Sent successfully through telematics

Section 2.0

Why OBECM Exists

2.1

Regulatory and Analytical Purpose

OBECM exists because the vehicle must collect certain energy-consumption-related data and make it available for downstream analysis at levels such as:

  • Fleet level
  • Model-year level
  • Vehicle-line level

For BEV applications, OBECM is focused on energy-use behavior rather than the broader and more complex monitoring scope of OBM. The main idea is that the vehicle keeps track of important cumulative energy-related values and sends those values out at defined intervals.

2.2

Snapshot-Based Behavior

A key part of understanding OBECM is knowing that it works more like a snapshot than a full history report.

🔄
Vehicle keeps updating
total values internally
→
📸
OBECM takes a
picture of those values
→
☁️
Snapshot sent out
through telematics
💡
Closer to an Odometer Than a Trip History

OBECM is closer to an odometer-style reporting feature than a full trip-history packaging system.

2.3

What the Vehicle Sends

For OBECM, the vehicle sends current cumulative values at the time of transmission. It does not mainly calculate monthly differences inside the vehicle. For example, the vehicle may keep track of:

  • Total distance traveled
  • Total energy used
  • Total energy charged
⚠️
Monthly Differences Are Calculated Externally

If an external system wants to know what changed between one month and the next, that difference is calculated outside the vehicle by comparing one snapshot to the next.

Section 4.0

Main Functions of OBECM

4.1

Data Collection

OBECM collects specific energy-consumption-related values. For BEV applications, this generally includes:

  • Total distance traveled
  • Vehicle speed-related data
  • Total energy charged into the battery
  • Battery current
  • Battery voltage
  • State of charge
  • Discharge-related energy values

The exact applicable list depends on the program and requirement set in use.

4.2

Data Structuring

OBECM organizes the collected values into a defined data set for reporting — narrower and more focused than OBM's larger layered packages.

Signals / Values
→
OBECM Snapshot
→
Diagnostics / Off-Board
4.3

Diagnostic Support

OBECM data must be available locally through diagnostics. This allows diagnostic tools connected through the OBD path to read the relevant data from the required ECUs.

4.4

Off-Board Support

OBECM must support sending the final data set out through the telematics path. The OBECM data is sent on a periodic basis — a monthly off-board schedule is used to improve the likelihood that data is received for each vehicle over time.

4.5

Display Support

OBECM also includes a small HMI-related element. A value related to OBECM is displayed in the center stack display through APIM. This is only a small display subset, not the full OBECM data set.

Section 6.0

ECU Roles in OBECM

🧠
PCM
Central ECU

Receives relevant inputs, supports the OBECM data set, and makes required values available for diagnostics and off-board. Central for vehicle-level values such as distance.

⚡
BECM
Energy Source

Especially important for BEV applications. Generates battery-related values, supports battery diagnostic readout, and provides battery data needed for the OBECM set.

🔀
ECG
Network Gateway

Helps messages move where they need to go. Not the main logic owner, not the cloud sender. ECG helps data travel — it does not create or own the OBECM data set.

📡
TCU
Cloud Sender

Receives the final OBECM payload and sends it to the cloud. If correct data exists in source modules but TCU does not send it, OBECM is not complete end to end.

🖥️
APIM
Display Path

Supports the center display. A limited OBECM-related value is displayed through the center stack. This is a display feature, not the full data set.

⚠️
IPC
Not Main Path

Not a main OBECM path in the described BEV application. Unlike some broader feature boundaries, OBECM does not rely on IPC for the described BEV use case.

Section 7.0

End-to-End OBECM Flow

7.1

Normal Flow

  1. The customer uses the vehicle normally.
  2. The vehicle updates energy-related values.
  3. Source ECUs such as PCM and BECM hold the required values.
  4. Data moves across the network where needed.
  5. The OBECM data set is formed from the required values.
  6. The data becomes available through diagnostics.
  7. The OBECM payload is sent toward TCU.
  8. TCU sends the payload to the cloud.
  9. A limited value may also be sent to APIM for display.
7.2

Why End-to-End Matters

⚠️
OBECM can fail at multiple points

A required value may never be available. A value may exist but not reach the needed destination. A correct data set may exist but never leave through TCU. OBECM should always be treated as an end-to-end system, even though it is simpler than OBM.

Main Output Paths

🔌
8.1 Diagnostic Path

OBD-connected tools read relevant values from PCM and BECM through the OBD path.

☁️
8.2 Cloud Path

8.3 HMI Path

Section 9.0

OBECM in BEV Applications

9.1

BEV Relevance

OBECM is especially relevant in BEV applications because it captures energy-consumption-related behavior and battery-related values. For BEVs, this includes values related to:

  • Total energy charged
  • Battery current
  • Battery voltage
  • State of charge
  • Discharge-related energy behavior
9.2

Practical BEV Focus Areas

  • Diagnostic readout path
  • Cloud off-board path
  • Center display value
  • Battery-related values from BECM
  • Correct data handling between PCM, BECM, ECG, and TCU

Section 10.0

OBECM Compared to OBM

10.1

Relationship

🔗
Related — Not the Same

OBECM and OBM are related features, but they are not the same. They differ significantly in scope, complexity, and packaging behavior.

10.2 / 10.3

Difference in Scope

OBECM Simpler · Narrower
  • Energy-consumption-related values
  • Cumulative totals
  • Periodic snapshot reporting
  • Focused BEV energy scope
OBM Broader · Deeper
  • Broader monitoring scope
  • Trip-based packaging
  • Lifetime and structured monitored data
  • Larger, more layered packages

Section 12.0

Why the Message List Matters

12.1

Signal Dependency

Even though OBECM is simpler than OBM, it still depends on the required signals and values being available. If the needed values are not added correctly to the network and the receiving ECUs do not have access to them, the feature cannot function properly.

12.2

Ownership and DCR Actions

Engineers still need to understand TX ECU ownership, RX ECU ownership, and required DCR actions. Without correct signal availability and routing, deeper validation cannot begin.

⚠️
Blocker Risk

If signals do not move from source to receiver, the feature is blocked before deeper validation can begin — even in a simpler feature like OBECM.

Section 15.0

Common Misunderstandings

❌ Misconception 15.1
"OBECM is the same as OBM."

OBECM is related to OBM but is not the same feature. OBECM is smaller in scope and simpler in behavior.

❌ Misconception 15.2
"OBECM builds trip-by-trip packages."

OBECM does not mainly behave like a large trip-by-trip packaging system. It is focused on cumulative values and periodic snapshots.

❌ Misconception 15.3
"ECG owns the feature data."

ECG helps messages move across the network. It does not own the main feature logic.

❌ Misconception 15.4
"TCU is the OBECM brain."

TCU mainly sends the data out of the vehicle. It does not create the OBECM logic or data model.

❌ Misconception 15.5
"The vehicle calculates monthly differences internally."

The vehicle mainly sends current cumulative values at the time of transmission. Monthly differences are calculated externally by comparing snapshots.

Section 16.0

Recommended Learning Path

16.1

Suggested Order

Step 1
Understand that OBECM is a periodic snapshot reporting feature

It sends current cumulative values at the time of transmission, not a trip-by-trip history.

Step 2
Learn the roles of PCM, BECM, ECG, TCU, and APIM

Understand what each ECU contributes and what it is not responsible for.

Step 3
Understand the difference between cumulative totals and monthly differences

The vehicle tracks totals; external systems derive period-over-period differences.

Step 4
Learn the three main output paths

Diagnostics, cloud, and HMI — each serves a different purpose.

Step 5
Review applicable requirements and signal ownership

Understand TX/RX ownership and required DCR actions for the applicable signals.

Step 6
Review the boundary diagram and message list

Tackle detailed requirement items after architecture and system flow are clear.

16.2

Why This Order Works

✅
Architecture First

Once the high-level OBECM behavior is clear, the detailed requirements become much easier to understand. Starting with architecture and system flow is usually more effective than starting with a long requirement list.

Summary

Final Summary

OBECM is a vehicle data feature that collects energy-consumption-related data, organizes it into a defined data set, makes it available through diagnostics, and sends it to the cloud through telematics.


In a BEV application:

  • ›PCM and BECM provide the required values
  • ›ECG helps messages move where they need to go
  • ›TCU sends the final data set to the cloud
  • ›APIM displays a limited related value on the center screen

OBECM is successful only when the data is: applicable · present · routed correctly · readable · structured correctly · sent successfully
