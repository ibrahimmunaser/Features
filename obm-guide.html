Section 1.0

On-Board Monitoring (OBM)

1.1

Overview

On-Board Monitoring, or OBM, is a vehicle feature that collects required operating data, organizes it into structured data sets, stores it, makes it available through diagnostics, and sends it to the cloud through telematics.

💡
Simple Definition

OBM is a vehicle data system that monitors how the vehicle is being used and turns that information into structured reports.

The main purpose of OBM is to support the collection, storage, and off-boarding of regulated vehicle-use data. This data can be accessed locally through diagnostics or sent externally through telematics for broader analysis.

For battery electric vehicles, OBM is especially focused on:

  • Charging behavior
  • Battery behavior
  • Trip-related information
  • Long-term vehicle-use statistics
ℹ️
Customer Transparency

OBM is not a feature the customer manually turns on or operates directly. The customer simply uses the vehicle normally — driving, charging, stopping, and parking all create the operating conditions that generate OBM data.

OBM is also not the main owner of the scientific correctness of every value inside the data package. Deep domain correctness — such as detailed battery-model behavior or emissions-model behavior — is generally owned by the relevant powertrain or battery teams. OBM's main role is to ensure that the required data is present, structured correctly, stored correctly, available through the correct paths, and sent successfully.

Section 2.0

Why OBM Exists

2.1

Regulatory and Analytical Purpose

OBM exists because the vehicle must collect certain regulated operating data and make it available for downstream analysis. This data is not mainly intended to monitor one specific customer. Instead, it supports analysis at a broader level:

  • Fleet level
  • Model-year level
  • Vehicle-line level

For battery electric vehicles, the focus is more heavily on battery behavior, charging behavior, trip-related energy use, and long-term vehicle-use patterns.

ℹ️
Structured Package

The vehicle does not necessarily send every internal detail it knows. Instead, it sends the required structured package that represents the applicable data for the program and powertrain.

Section 3.0

High-Level Concept

3.1

OBM as a Data Pipeline

The easiest way to understand OBM is to view it as a data pipeline inside the vehicle. The high-level flow is:

Vehicle Behavior
→
ECU Signals
→
PCM Processing
→
OBM Package
→
Diagnostics / Cloud
  • The customer uses the vehicle normally.
  • Multiple ECUs generate or provide the required data.
  • The PCM performs the main OBM logic.
  • The data is grouped into structured OBM data sets.
  • The data is made available through diagnostics and telematics.
3.2

Why This Matters

⚠️
OBM is multi-step — failure can happen anywhere

A signal can exist in the vehicle and OBM can still fail if: the signal never reaches the correct ECU; the signal is not inserted into the correct data structure; the final package is not sent successfully.

Section 4.0

Main Functions of OBM

4.1

Data Collection

OBM collects many different values from the vehicle. These can include trip-related values, lifetime values, battery-related values, and metadata and identification values. The exact set depends on applicable requirements and vehicle powertrain.

4.2

Data Structuring

OBM does not simply store raw signals in one flat list. It groups data into structured sets, working in layers:

Signals
→
Trip Data
→
OBM Package
→
Storage / Off-Board
4.3

Diagnostic Support

OBM data must be available locally through diagnostics. This allows tools connected through the OBD path to read the relevant data from the appropriate ECUs.

4.4

Off-Board Support

OBM must support sending the final data package out of the vehicle through the telematics path. It is not enough for the data to be built correctly — the system must also support successful delivery to the cloud so the package is not silently lost in transit.

4.5

Display Support

A limited subset of OBM-related information may also be shown in the vehicle HMI, depending on program and powertrain applicability.

Section 6.0

ECU Roles in OBM

🧠
PCM
Primary Brain

Main OBM logic center. Receives inputs from all other ECUs, runs OBM logic, builds structured data sets, and prepares the final OBM package.

⚡
BECM
Battery Source

Source for battery and high-voltage data elements in BEV applications. Provides battery-related values to both diagnostics and PCM for packaging.

🔀
ECG
Network Gateway

Routes messages between ECUs and networks. Does not own the main feature logic or build data sets. Helps data travel — it does not build it.

📡
TCU
Cloud Sender

Telematics sender. Receives the final off-board payload and sends it to the cloud. If TCU does not send successfully, OBM is not complete end to end.

🖥️
APIM
Display Path

Supports the center display. Shows a limited subset of OBM-related information on the HMI. This is a display subset only — not the full OBM package.

⚠️
IPC
Warning Path

Supports warning behavior in some non-BEV applications. For BEV applications this warning path may not apply, though IPC may still appear in a generic feature boundary.

Section 7.0

End-to-End OBM Flow

7.1

Normal Flow

  1. The customer uses the vehicle normally.
  2. The vehicle generates operating data.
  3. Source ECUs (such as BECM) create the needed values.
  4. Data moves across the network where needed.
  5. PCM receives the required inputs.
  6. PCM performs the main OBM logic.
  7. PCM builds the required data structures.
  8. The data becomes available through diagnostics.
  9. PCM sends the off-board package toward TCU.
  10. TCU sends the package to the cloud.
  11. A limited subset may also be sent to APIM for display.
7.2

Why End-to-End Matters

⚠️
OBM can fail at several points in the chain

A signal may exist but never reach PCM. A signal may reach PCM but never be packaged correctly. A correct package may be built but never leave through TCU. That is why OBM should always be understood as an end-to-end system.

Section 8.0

Main Output Paths

🔌
8.1 Diagnostic Path

OBD-connected tools can read relevant values from PCM and BECM directly.

☁️
8.2 Cloud Path

Final OBM package sent to the cloud via TCU. The main off-board path for OBM.

🖥️
8.3 HMI Path

Limited subset displayed in center screen through APIM. Not the full OBM package.

Section 9.0

OBM in BEV Applications

9.1

Powertrain Applicability

🎯
Applicability Must Always Be Considered

For a BEV, not every item in a full global OBM requirement list will apply. Some requirements are intended for combustion-related systems and do not apply to a battery electric vehicle. Engineers should focus on what is applicable to the specific program and powertrain.

9.2

Practical BEV Focus Areas

  • Center display path
  • Diagnostic readout path
  • Cloud off-board path
  • Battery and charging-related data
  • Correct PCM packaging of the applicable data set

Section 10.0

What Engineers Validate in OBM

10.1

Main Validation Focus

The main engineering job in OBM is not to prove the scientific truth of every number. The main job is to ensure that the data system works correctly.

10.2

Typical Validation Areas

✓
The required signals exist
✓
The signals reach the correct ECUs
✓
The PCM builds the correct data structures
✓
The data is readable through diagnostics
✓
The display path works where applicable
✓
The off-board package reaches the cloud successfully
✓
Correct requirements are applied for the correct powertrain and program
10.3

Key Takeaway

✅
OBM Engineering is About

Data flow · Structure · Routing · Availability · Delivery

Section 11.0

Why the Message List Matters

11.1

Signal Dependency

Signal and message-list work is one of the highest-priority parts of OBM. Signals may be:

  • Shared signals
  • Related-feature signals
  • OBM-specific signals
⚠️
Blocker Risk

If the required signals are not added correctly, the rest of the feature cannot function properly.

11.2

Ownership and DCR Actions

Engineers must understand TX ECU ownership, RX ECU ownership, and required DCR actions. If the signals do not move from source to receiver, the feature is blocked before deeper validation can begin.

Section 14.0

Common Misunderstandings

❌ Misconception 14.1
"OBM is just one signal."

OBM is a full data system, not one signal or one message. It is a multi-step pipeline spanning multiple ECUs and output paths.

❌ Misconception 14.2
"ECG owns the feature data."

ECG mainly supports message movement across the network. It does not own the main feature logic or build the data set.

❌ Misconception 14.3
"TCU is the OBM brain."

TCU mainly sends. It does not build the OBM data package. The PCM is the feature brain.

❌ Misconception 14.4
"All global requirements apply equally."

Not every requirement in a large global document applies equally to every powertrain. Applicability must always be considered for the specific program.

❌ Misconception 14.5
"OBM and OBECM are the same."

OBM and OBECM are related, but they are not the same feature. OBM is broader and more complex.

Section 15.0

Recommended Learning Path

15.1

Suggested Order

Step 1
Understand the full end-to-end flow

Start with the big picture before diving into ECU details or requirements lists.

Step 2
Learn ECU roles: PCM, BECM, ECG, TCU, APIM, IPC

Understand what each ECU is responsible for and what it is not.

Step 3
Understand signal presence vs. package correctness

A signal existing in the vehicle does not mean it is being packaged correctly.

Step 4
Learn the three main output paths

Diagnostics, cloud, and HMI — each serves a different purpose.

Step 5
Learn which requirements apply to the target powertrain

BEV applicability differs from ICE. Never assume full global applicability.

Step 6
Review boundary diagram and message list

Tackle the detailed requirement items after the architecture is clear.

15.2

Why This Order Works

✅
Architecture First

Starting with the architecture usually makes more sense than starting with a large requirement list. Once the system flow is clear, the detailed requirements become much easier to understand.

Section 16.0

Summary

Final Summary

OBM is a vehicle data feature that collects required operating data, organizes it into structured data sets, makes it available through diagnostics, and sends it to the cloud through telematics.


In a BEV application:

  • ›Source ECUs create the data
  • ›PCM acts as the main OBM brain
  • ›BECM provides major battery-related data
  • ›ECG helps messages move where they need to go
  • ›TCU sends the final package to the cloud
  • ›APIM can show a limited subset on the display

OBM is successful only when the data is: applicable · present · routed correctly · packaged correctly · readable · sent successfully
